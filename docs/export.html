Export App

Online step — ingest documents, build a knowledge bundle, download a self-contained zip. Runs entirely in the browser. No backend, no uploads, no API keys.

What happens during export

1

Parse documents

PDF, DOCX, Markdown, and plain text files are parsed to plain text inside a Web Worker using pdfjs-dist and mammoth. The main thread never touches raw file bytes — parsing is fully off-thread.

2

Chunk and embed

Text is split into 100-token chunks with a 20-token overlap. 100 tokens is intentionally small — it fits comfortably in the 4K context window of Phi-3-mini and Gemma-2B alongside the question and answer.

Each chunk is embedded by bge-small-en-v1.5-q8 running via ONNX inside Transformers.js in a dedicated embed worker. Vectors are 384-dimensional and L2-normalised (cosine similarity = dot product).

3

Build the knowledge graph

Cosine similarity (dot product, since vectors are already normalised) is computed between all chunk pairs. Pairs scoring ≥ 0.75 become edges. This reuses the vectors just computed — zero extra LLM calls.

graph.json — { "version": "1.0", "edges": [{ "a": "chunk_0001", "b": "chunk_0047", "score": 0.81 }] }

At query time, a 2-hop BFS from the RAG hit chunk IDs expands the retrieval neighborhood across semantically connected content.

4

Build the search index

All chunks + their vectors are inserted into an Orama BM25 + vector hybrid index. The index is serialised and gzip-compressed to orama-index.json.gz.

5

Capture the embed model cache

After Transformers.js loads bge-small-en-v1.5, its Cache API entries (the ONNX weights, ~24 MB) are drained and bundled into the zip as embed-cache/. On import, these are restored to Cache API under their original HuggingFace CDN URLs — Transformers.js finds them and makes zero network requests.

6

Fetch chat model weights

A zip worker fetches the chat model shards (safetensors) and compiled WebGPU WASM kernel directly from the WebLLM CDN. They are streamed straight into the zip without going through main-thread RAM. The model is never fully in memory.

7

Stream-compress to disk

Everything is assembled by fflate in streaming mode and written chunk-by-chunk to the user's chosen file via the File System Access API (showSaveFilePicker). The full zip is never held in RAM — even for a 1.3 GB bundle.

Technical constraints

⚡

GitHub Pages 100 MB file limit: model shards are fetched from the WebLLM CDN at export time, not stored in the repo. The zip assembly happens in the browser.

💾

Never buffer the full zip in RAM: fflate streams each file in as a chunk is ready and writes it immediately via FileSystemWritableFileStream.

🔀

Three off-thread workers: document-worker (parse), embed-worker (embed + cache capture), zip-worker (compress + model fetch). Main thread only manages progress UI.

🧠

SharedArrayBuffer required: COOP/COEP headers are provided by coi-serviceworker — the service worker installs on first load and reloads the page to activate the cross-origin isolation headers.
