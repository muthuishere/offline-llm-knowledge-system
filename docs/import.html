Import App

Offline forever — drop the zip into any browser, wait for OPFS extraction, then chat with the knowledge base. No internet required after the first load.

What happens during import

1

Stream-extract to OPFS

An unzip worker reads the zip as a ReadableStream — the full file is never buffered in RAM. fflate decompresses each entry and writes it to the browser's Origin Private File System (OPFS), a persistent, sandboxed storage that survives page reloads.

Extraction is keyed by manifest_hash. Dropping the same zip a second time finds the hash, skips extraction, and loads directly from OPFS.

2

Restore the embed model cache

The embed-cache/ entries from the zip are written back to the browser's Cache API under their original HuggingFace CDN URLs. Transformers.js finds them on first request and makes zero network calls — the embedding model loads entirely offline.

3

Load search index and knowledge graph

chunks.json and graph.json are read from OPFS. Chunks are inserted into a fresh Orama index for hybrid search. The graph edges are loaded and an in-memory adjacency map is built — a plain Map<chunkId, chunkId[]>, no graph library needed.

4

Load the chat model from blob URLs

Model safetensors shards are read from OPFS into Blob objects, converted to blob: URLs, and injected into WebLLM's appConfig. The engine loads entirely from local memory — no CDN, no cache, no network. Blob URLs are revoked after engine init to free memory.

5

Query pipeline — per message

Each question triggers the full three-layer retrieval pipeline before the model generates an answer:

1. Embed query → 384-dim vector (Transformers.js, off-thread)
2. Hybrid search → BM25 + vector in Orama → top-3 RAG chunks
3. Graph BFS → 2-hop walk from RAG chunk IDs → semantic neighborhood
4. Wiki lookup → OPFS cache hit / miss + WebLLM synthesis
5. Assemble user-turn context → wiki → graph → RAG
6. Stream response from WebLLM

All context goes in the user turn, not the system prompt. Small models (Phi-3-mini, Gemma-2B) attend to user-turn content far more reliably than distant system content — classic RAG mistake is putting context in system, which produces vague answers.

Lazy wiki — how it compounds

?

First query on a document

Wiki cache is empty. The model synthesizes a concise 3–5 sentence reference page for the top-1 source document, using the RAG context as input. The result is written to opfs/wiki/{slug}.md.

✓

Subsequent queries on the same document

Cache hit — the wiki page is read from OPFS and placed first in the context window. No model call. The wiki advantage compounds with use, exactly as Karpathy described — knowledge is pre-synthesized, not re-derived on every query.

Technical constraints

📁

OPFS persistence: extraction is idempotent — keyed by manifest_hash. The same zip dropped twice skips extraction and loads from OPFS in seconds instead of minutes.

🔗

Blob URL lifecycle: model shard blob URLs are revoked immediately after the WebLLM engine finishes loading. Holding them open would prevent the buffers from being garbage-collected.

🧠

SharedArrayBuffer required: COOP/COEP isolation headers are required for WebLLM's WASM threading. Provided by coi-serviceworker — it installs on first visit and reloads the page. State is saved to sessionStorage before reload.

🤏

Small model quality ceiling: wiki synthesis and RAG answers are bounded by the model (TinyLlama / Gemma-2B / Phi-3-mini). The lazy wiki is contextualised at query time, which is meaningfully better than a blind export-time pass with the same model.
